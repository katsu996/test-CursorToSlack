# 保守・改善バックログ（優先順位付き）

機能追加や改修を続けるときに、**再発防止・運用負荷・互換性**の観点で後から効く作業をまとめています。優先度は次の意味です。

| 優先度 | 意味 |
|--------|------|
| **P0** | 放置すると障害・セキュリティ・beatoraja 互換破壊につながりやすい |
| **P1** | 回帰を防ぐためのテストや CI。変更のたびに手戻りが出やすい |
| **P2** | 可読性・分割・設定の整理。長期的な改修コストを下げる |
| **P3** | UX やドキュメント、あればよいが必須ではない |

---

## P0（最優先）

1. **beatoraja / jbmstable-parser との互換の回帰防止**  
   - 本体やパーサの `decodeJSONTableData`・`TableData.validate()` の前提（`level` / ハッシュ長 / 空 `course` など）が変わったときに気づけるよう、**期待する入出力のスナップショット**をドキュメントまたはテストで固定する。  
   - リリースノートや upstream の差分を、表 JSON の生成ロジック変更のトリガーとして扱う。

2. **`sql_where` の安全性と運用ガード**  
   - 現状は括弧内に文字列連結しているため、**信頼できる設定ファイルのみ**をコミットする運用が前提。  
   - 可能なら **許可リスト方式**（例: プリセット名と固定 SQL の対応）や、**列名・演算子のホワイトリスト**で構造化条件から SQL を生成する道を検討する（自由記述と併用でも可）。

3. **外部難易度表 URL 障害時の挙動**  
   - 元表ホストのダウン・タイムアウト・TLS 失敗で Actions が赤くなる。**リトライ・タイムアウト秒数・失敗時の明確なログ**を仕様として決め、ワークフロー上で原因が追いやすいメッセージに揃える。

4. **`filtered_data.json` が 0 件のとき**  
   - beatoraja は `TableData.validate()` で失敗する。**CI で警告を目立たせる**（ジョブサマリ、`continue-on-error` ではなく明示的な exit 方針の文書化）か、**空のときはデプロイを止める／プレースホルダを出す**など、運用ポリシーを README と揃える。

---

## P1（テスト・CI）

1. **`tools/table-filter/` のユニットテスト（標準ライブラリのみ）**  
   - 相対 `data_url` の解決、HTML `bmstable` からのヘッダー解決、複数ヘッダーのマージと重複除去、`custom_level_mapping`、beatoraja 向け行のサニタイズ（`filtered_data` vs `filtered_data_enriched`）、空 `course` のヘッダー整形、`level_stats.json` の version 2 形式など、**純関数に近い部分から**切り出してテストする。

2. **GitHub Actions での `ruff check`（任意で `format`）**  
   - ローカルと CI で同じコマンドにし、PR ごとにスタイルと明らかなバグを検知する。

3. **ワークフローと Python バージョンの明示**  
   - `actions/setup-python` のバージョンと `AGENTS.md` の想定（例: 3.12）を一致させ、メジャー更新時にだけ意識的に追従する。

4. **生成物のスモークテスト**  
   - デプロイ前に `filtered_header.json` の JSON パース、`data_url` の相対性、`filtered_data.json` が配列であること、**0 件なら policy に従った終了コード**などを軽く検証するスクリプトを足す。

---

## P2（構造・保守性）

1. **`filter_table.py` の責務分割**  
   - 取得・マージ・サニタイズ・書き出しをモジュールまたは関数群に分け、変更箇所の探索コストを下げる。

2. **設定スキーマの文書化**  
   - `filter_config.json` のキーを一覧化し（必須・任意・非推奨）、**例は `filter_config.example.json` と同期**するチェックを README か CI に書く。

3. **`docs/index.html` / `level-stats.html` の共通化**  
   - CSS 変数・テーマ切替・カードスタイルが二重になっているため、**共通の `docs/assets/` 以下の CSS または小さな JS** に寄せて重複修正を減らす。

4. **`output_header_name` と Pages 用タイトルの分離**  
   - beatoraja 内では短い表名、サイトでは長い表名、といった要件が出たときのため **`page_title` 専用キー**（未設定時は `output_header_name` にフォールバック）を検討する。

5. **`songdata.db` のサイズ増大**  
   - Git 履歴が肥大化したら **Git LFS** や「DB は別アーティファクトで渡す」構成の再検討（[data/README.md](data/README.md) と整合）。

---

## P3（UX・ドキュメント・任意）

1. **アクセシビリティ**  
   - 折りたたみパネル・テーマ切替に `aria-expanded` 等を付与し、キーボード操作を確認する。

2. **クライアント側の軽いバリデーション**  
   - `browser_rows.json` の `meta` 欠損時のメッセージ統一、古い `level_stats.json`（version 1）の表示分岐の整理。

3. **別フロントエンド化の判断材料**  
   - 表・フィルタ UI がさらに複雑になるなら、ビルド付き（Vite + React + shadcn/ui 本体）への移行可否を **コスト表**で一度だけ整理する（現状は静的 HTML で運用コストが低い利点がある）。

4. **コミュニティ向けの「よくある質問」**  
   - Table URL の末尾スラッシュ、`bmstable` meta、0 件エラーなど、README §8〜9 と重複しない粒度で FAQ を `docs/` に切り出す。

---

## 定期メンテ（カレンダーではなくトリガーで実施）

- **beatoraja / LR2oraja の新バージョン**が出たら: 難易度表まわりの挙動を確認し、必要なら P0 のスナップショットを更新する。  
- **`source_header_urls` を差し替えたとき**: 元表の JSON キー変更（`level` 以外の難易度列など）で `custom_level_source_key` や level 集計がずれていないか確認する。  
- **GitHub の Actions / Pages の仕様変更**: `pages.yml` の `actions/*` メジャー更新時はリリースノートを読んでからマージする。
